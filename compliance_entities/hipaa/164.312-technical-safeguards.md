# §164.312 — Technical Safeguards

**Registry path:** `/regulation-registry/HIPAA/164.312/`
**Version:** Current (45 CFR Part 164 Subpart C)
**Last parsed:** 2026-05-20
**Applies to:** Covered entities (healthcare providers who transmit health information electronically, health plans, healthcare clearinghouses) and their business associates who create, receive, maintain, or transmit PHI on behalf of a covered entity
**Trigger:** Creation, receipt, maintenance, or transmission of Protected Health Information (PHI) in connection with a covered function; business associate agreements (BAAs) flow down requirements to subcontractors
**Jurisdiction:** United States; extraterritorial reach for foreign entities handling PHI of US patients or providing services to US covered entities
**Not applicable to:** Life insurers outside healthcare coverage; employers' employment records; education records covered by FERPA; de-identified data meeting Safe Harbor or Expert Determination standard; purely personal health records with no commercial healthcare function
**Overall confidence:** MEDIUM — unique user ID and access control existence are DETERMINISTIC; encryption, logoff, and integrity methods are PARAMETERIZED
**R = Required | A = Addressable**

---

## Scope summary

§164.312 defines the technical controls required to protect ePHI at rest and in transit. It is where most of the security architecture sits: access control, audit logging, integrity controls, authentication, and encryption. Critically, **encryption is addressable under HIPAA** — but the HITECH safe harbor for breaches creates enormous practical pressure to encrypt regardless. This file treats encryption as effectively required in any modern environment and documents that assumption explicitly.

The unique user identification requirement is the most-cited §164.312 violation in OCR audits. Shared accounts for ePHI access are a bright-line prohibition.

### (a)(2)(iv) — Encryption and Decryption (A — PARAMETERIZED)

### Source excerpt

> *Implement a mechanism to encrypt and decrypt electronic protected health information.*

### Element extraction

| Element | Value | Classification |
|---|---|---|
| Subject | ePHI at rest on all covered systems | DETERMINISTIC |
| Condition | Addressable — but HITECH breach safe harbor makes non-encryption a high-risk posture | DETERMINISTIC |
| Obligation | Encryption implemented for ePHI at rest; OR documented rationale for non-implementation with equivalent alternative. If implemented: uses NIST-approved algorithm, key management documented | PARAMETERIZED |

"""
164.312(a)(2)(i) — Unique User Identification
Confidence: HIGH | Human Review: NOT REQUIRED
"""


def test_no_shared_accounts_on_ephi_systems(user_accounts):
    shared = [
        a for a in user_accounts
        if a.get("ephi_system_access") and a.get("shared_account")
    ]
    assert not shared, (
        f"CRITICAL VIOLATION: {len(shared)} shared account(s) with ePHI access "
        f"(prohibited by §164.312(a)(2)(i)): "
        f"{[a['account_id'] for a in shared]}"
    )


def test_all_ephi_user_ids_are_unique(user_accounts):
    ephi_accounts = [a for a in user_accounts if a.get("ephi_system_access")]
    ids = [a["user_id"] for a in ephi_accounts]
    duplicates = [uid for uid in ids if ids.count(uid) > 1]
    assert not duplicates, (
        f"CRITICAL VIOLATION: Duplicate user IDs found across ePHI systems: "
        f"{list(set(duplicates))}. Each user must have a unique identifier."
    )


def test_no_generic_accounts_on_ephi_systems(user_accounts):
    generic_patterns = {"admin", "administrator", "root", "guest", "test",
                        "default", "user", "service", "shared"}
    violations = [
        a for a in user_accounts
        if a.get("ephi_system_access")
        and a.get("user_id", "").lower() in generic_patterns
    ]
    assert not violations, (
        f"WARNING: {len(violations)} account(s) with generic names have ePHI access "
        f"— verify these are not shared accounts: "
        f"{[a['account_id'] for a in violations]}"
    )

import pytest

MIN_TLS_VERSION = "1.2"
PROHIBITED_CIPHER_PATTERNS = {"RC4", "DES", "3DES", "NULL", "EXPORT", "anon", "ADH", "AECDH"}


@pytest.mark.assumption(
    id="ASSUME-312-007",
    description=(
        "Adequate: TLS 1.2+, AEAD cipher suites; TLS 1.0/1.1 and weak ciphers disabled; "
        "valid CA cert. Self-signed permitted only on private networks with justification."

    approved_by="Compliance Officer",
    review_date="2026-05-20",
)
def test_all_ephi_transmissions_use_minimum_tls(transmission_configs):
    def tls_version_ok(version_str):
        try:
            return float(version_str) >= float(MIN_TLS_VERSION)
        except (TypeError, ValueError):
            return False

    violations = [
        c for c in transmission_configs
        if c.get("transmits_ephi")
        and not tls_version_ok(c.get("min_tls_version"))
        and not c.get("non_implementation_rationale")
    ]
    assert not violations, (
        f"VIOLATION: {len(violations)} ePHI transmission endpoint(s) below TLS "
        f"{MIN_TLS_VERSION} without documented rationale: "
        f"{[c['system_id'] for c in violations]}"
    )


@pytest.mark.assumption(id="ASSUME-312-007")
def test_no_weak_cipher_suites_on_ephi_endpoints(transmission_configs):
    violations = []
    for config in transmission_configs:
        if not config.get("transmits_ephi"):
            continue
        enabled_ciphers = config.get("enabled_cipher_suites", [])
        weak = [
            c for c in enabled_ciphers
            if any(p in c for p in PROHIBITED_CIPHER_PATTERNS)
        ]
        if weak:
            violations.append(
                f"System {config['system_id']}: weak cipher(s) enabled: {weak}"
            )
    assert not violations, (
        f"VIOLATION: {len(violations)} ePHI endpoint(s) with weak cipher suites:\n"
        + "\n".join(violations)
    )


@pytest.mark.assumption(id="ASSUME-312-007")
def test_tls_certificates_valid(transmission_configs):
    from datetime import datetime
    violations = []
    for config in transmission_configs:

## Notes for the registry

- **Encryption addressable status — practical reality:** While §164.312(a)(2)(iv) and §164.312(e)(2)(ii) both list encryption as addressable, HHS has stated in guidance that "the use of encryption is strongly encouraged" and OCR invariably scrutinizes absent encryption in breach investigations. Treat encryption as de facto required for any patient-facing or internet-connected system; reserve "addressable and not implemented" only for air-gapped clinical systems with no external connectivity.
- **FIPS 140-2/140-3 validation:** HHS guidance references NIST standards for encryption. FIPS 140-2 validated cryptographic modules are the expected baseline for government-adjacent healthcare organizations (HRSA grantees, VA contractors, etc.).
- **Session tokens and cookie security:** Automatic logoff (a)(2)(iii) should be paired with secure session management — HttpOnly, Secure, SameSite cookie flags for web applications accessing ePHI. These are not explicitly named in the rule but are implied by the access control and transmission security standards.
- **Log integrity and SIEM:** The tamper-evident log requirement in the assumption for (b) is best satisfied by forwarding logs to an immutable SIEM or write-once log store immediately upon generation. Local log storage on the same system being audited is inadequate.
