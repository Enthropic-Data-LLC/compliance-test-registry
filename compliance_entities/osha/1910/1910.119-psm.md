## Scope summary

1910.119 applies to processes that involve certain highly hazardous chemicals at or above threshold quantities (TQs) listed in Appendix A. It establishes 14 elements of a Process Safety Management system. The standard is extensive; this file covers the eight most automation-relevant elements. Elements heavily dependent on engineering judgment (e.g., the PHA methodology itself) are decomposed but result in Pattern 2 or Pattern 3 tests.

Facilities covered include chemical plants, refineries, petrochemical operations, and any facility with a covered HHC above TQ.

## (f) — Process Hazard Analysis (PHA)

### Source excerpt

> *The employer shall perform an initial process hazard analysis (hazard evaluation) on processes covered by this standard ... The employer shall use one or more of the following methodologies that are appropriate to determine and evaluate the hazards of the process: What-If; Checklist; What-If/Checklist; Hazard and Operability Study (HAZOP); Failure Mode and Effects Analysis (FMEA); Fault Tree Analysis; or An appropriate equivalent methodology.*

### Element extraction

| Element | Value | Classification |
|---|---|---|
| Subject | Each PSM-covered process | DETERMINISTIC |
| Condition | At initial implementation; revalidation at least every 5 years | DETERMINISTIC |
| Obligation | PHA conducted using an acceptable methodology; covers process hazards, previous incidents, engineering and administrative controls, consequences of failure, human factors, qualitative evaluation of effects on employees; documented; recommendations tracked to resolution | PARAMETERIZED |
| Evidence | `pha_record.process_id`, `methodology`, `completion_date`, `revalidation_date <= completion_date + 1826 days`, `recommendations_tracked_to_close == true` | PARAMETERIZED |

**Assumption (ASSUME-119-002):** A PHA is deemed adequate when: (1) the methodology is one of the six enumerated or a documented equivalent; (2) a multi-disciplinary team is documented including operations, engineering, and safety representation; (3) all identified hazard scenarios have documented recommendations with responsible owners and target dates; (4) recommendations are resolved or risk-accepted with documented rationale.

## (m) — Management of Change (MOC)

### Source excerpt

> *The employer shall establish and implement written procedures to manage changes (except for "replacements in kind") to process chemicals, technology, equipment, and procedures; and changes to facilities that affect a covered process.*

### Element extraction

| Element | Value | Classification |
|---|---|---|
| Subject | Any change to a PSM-covered process (chemicals, technology, equipment, procedures) | PARAMETERIZED |
| Condition | Change is not a "replacement in kind" | PARAMETERIZED |
| Obligation | Written MOC procedure; each change documented with: technical basis, safety/health considerations, modifications to procedures, necessary time period, authorization prior to startup, training of affected employees | DETERMINISTIC |
| Evidence | `moc_record.change_id`, `technical_basis`, `safety_review`, `authorized_by`, `pre_startup_training_completed` | PARAMETERIZED |

**Assumption (ASSUME-119-003):** "Replacement in kind" is defined as a replacement that satisfies the design specifications of the replaced equipment or chemical in all respects (material, size, rating, function). Any deviation from a design specification — including a substitute chemical with different physical properties — is NOT replacement in kind and requires an MOC.

```python
"""
1910.119(g) — Operating Procedures
Confidence: HIGH | Human Review: NOT REQUIRED
"""
from datetime import datetime, timedelta

REQUIRED_PROCEDURE_TYPES = {
    "initial_startup",
    "normal_operations",
    "temporary_operations",
    "emergency_shutdown",
    "emergency_operations",
    "normal_shutdown",
    "startup_after_turnaround",
}

CERTIFICATION_INTERVAL_DAYS = 365


def test_operating_procedures_exist_for_all_processes(
    psm_processes, operating_procedures
):
    proc_process_ids = {p["process_id"] for p in operating_procedures}
    missing = [
        p for p in psm_processes
        if p["process_id"] not in proc_process_ids
    ]
    assert not missing, (
        f"VIOLATION: {len(missing)} PSM process(es) have no operating procedures: "
        f"{[p['process_id'] for p in missing]}"
    )


def test_operating_procedures_cover_all_required_types(operating_procedures):
    violations = []
    from itertools import groupby
    by_process = {}
    for proc in operating_procedures:
        by_process.setdefault(proc["process_id"], []).append(proc)
    for process_id, procs in by_process.items():
        types_present = {p["procedure_type"] for p in procs}
        missing = REQUIRED_PROCEDURE_TYPES - types_present
        if missing:
            violations.append(
                f"Process {process_id}: missing procedure types {missing}"
            )
    assert not violations, (
        f"VIOLATION: {len(violations)} process(es) missing required procedure types:\n"
        + "\n".join(violations)
    )


def test_operating_procedures_annually_certified(operating_procedures):
    violations = []
    by_process = {}
    for proc in operating_procedures:
        by_process.setdefault(proc["process_id"], []).append(proc)
    for process_id, procs in by_process.items():
        latest_cert = max(
            (p.get("annual_certification_date") for p in procs if p.get("annual_certification_date")),
            default=None,
        )
        if not latest_cert:
            violations.append(f"Process {process_id}: no annual procedure certification")
            continue
        age = datetime.utcnow().date() - latest_cert
        if age.days > CERTIFICATION_INTERVAL_DAYS:
            violations.append(
                f"Process {process_id}: annual certification {age.days} days old"
            )
    assert not violations, (
        f"VIOLATION: {len(violations)} process(es) with overdue procedure "
        f"certification:\n" + "\n".join(violations)
    )
```

```python
    assert not violations, (
        f"VIOLATION: {len(violations)} process(es) with overdue PHA revalidation:\n"
```

## Notes for the registry

- **RAGAGEP:** The equipment information requirement in (e)(3) references RAGAGEP (Recognized and Generally Accepted Good Engineering Practices). The applicable RAGAGEP for a given piece of equipment (e.g., ASME, API, NFPA) must be documented in the PSI. OSHA has cited facilities for failing to document RAGAGEP explicitly.
- **Pre-Startup Safety Review (PSSR):** (j) requires a PSSR before introducing HHC to a new or modified process. PSSR is closely linked to MOC — a test that verifies PSSR completion for all MOC records with a "new startup" flag is recommended.
- **Contractor requirements (i):** 1910.119(i) extends PSM obligations to contract workers. Contractor training records and safety program documentation should be included in the test suite.
- **RMP cross-reference:** Facilities subject to EPA's Risk Management Program (40 CFR Part 68) have overlapping but distinct requirements. PSI and PHA documentation may satisfy both, but the RMP submission process is separate.
