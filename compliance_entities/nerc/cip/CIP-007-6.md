# CIP-007-6 — System Security Management

**Registry path:** `/regulation-registry/NERC-CIP/CIP-007-6/`
**Version:** 6 (currently enforceable)
**Last parsed:** 2026-05-20
**Overall confidence:** MIXED — R2 (patch management) has the framework's canonical CONTESTED example; R1, R3, R4, R5 mostly DETERMINISTIC

## Scope summary

CIP-007 is the largest CIP standard by line count and the most operationally taxing. It covers ports/services, patch management, malicious code prevention, security event monitoring, and system access controls. Patch management (R2) is the canonical CONTESTED case the RDF framework was designed for — the "technically feasible" determination drives the framework's entire Pattern 2/3 split.

## R2 — Security Patch Management (the canonical case)

| Element | Value | Classification |
|---|---|---|
| Subject | Cyber Assets within H/M BCS | DETERMINISTIC |
| Condition | Source identified for patches; patch released; "technically feasible" to apply | PARAMETERIZED — "technically feasible" |
| Obligation | Within 35 calendar days: evaluate applicability; apply OR document mitigation plan OR revise mitigation timeframe | PARAMETERIZED — "mitigation plan" adequacy |
| Evidence | `patch_record.{evaluated_at, applied_at, mitigation_plan_id}` | CONTESTED — industry interpretation varies on mitigation adequacy |

## R3 — Malicious Code Prevention

| Element | Value | Classification |
|---|---|---|
| Subject | H/M Cyber Assets | DETERMINISTIC |
| Condition | Asset can run / accept executable code | DETERMINISTIC |
| Obligation | Deploy method(s) to deter, detect, or prevent malicious code; mitigate threat of detected malicious code; test/update signatures or methods | PARAMETERIZED — "method" intentionally implementation-agnostic |
| Evidence | `malicious_code_control.method`, `.last_signature_update_at`, `.detected_events[]` | PARAMETERIZED |

## R4 — Security Event Monitoring

| Element | Value | Classification |
|---|---|---|
| Subject | H/M Cyber Assets (logging at the asset and at EACMS) | DETERMINISTIC |
| Condition | Asset capable of generating logs | DETERMINISTIC |
| Obligation | Log events relating to detected successful login attempts, failed login attempts, malicious code; generate alerts for specified security events; review summaries or sampling at least every 15 calendar days; retain logs for last 90 calendar days | DETERMINISTIC |
| Evidence | `log_record.event_types_captured[]`, `.review_at`, `.retention_days`, `alert_record[]` | DETERMINISTIC |

## R5 — System Access Controls

| Element | Value | Classification |
|---|---|---|
| Subject | Interactive user access to H/M Cyber Assets | DETERMINISTIC |
| Condition | Account exists or login is attempted | DETERMINISTIC |
| Obligation | Identify and inventory accounts (R5.1); identify shared accounts and individuals authorized (R5.2); change default passwords (R5.4); enforce password length ≥ 8 / complexity from 3 of 4 character classes where technically feasible (R5.5); enforce password change at least every 15 months (R5.6); limit unsuccessful authentication attempts or alert (R5.7) | DETERMINISTIC for thresholds; PARAMETERIZED for "where technically feasible" |
| Evidence | `account_inventory[]`, `password_policy.{min_length, complexity, max_age_days}`, `auth_failure_threshold` | DETERMINISTIC |

### `tests/CIP-007/test_R2.py` (Pattern 2 + Pattern 3 — mixed)

```python
"""
CIP-007-6 R2 — Security Patch Management
Confidence: LOW (CONTESTED on mitigation plan adequacy)
Human Review: MANDATORY
"""
import pytest
from datetime import datetime, timedelta


@pytest.mark.assumption(
    id="ASSUME-007-001",
    description="Technically feasible = vendor patch + test env + change window + no caveat",
    approved_by="Compliance Officer",
    review_date="2026-01-15",
)
def test_CIP007_R2_patch_compliance(asset, patch_record):
    """R2.3: Patches applied within 35 days OR mitigation plan documented."""
    if is_technically_feasible(asset, patch_record):
        days = (patch_record.applied_at - patch_record.evaluated_at).days
        assert patch_record.applied_at is not None and days <= 35, (
            f"VIOLATION: {asset.asset_id} patch {patch_record.patch_id} "
            f"applied {days} days after evaluation (threshold 35)."
        )
    else:
        assert patch_record.mitigation_plan_id is not None, (
            f"VIOLATION: {asset.asset_id} patch {patch_record.patch_id} "
            f"deemed infeasible but no mitigation plan documented "
            f"(ASSUME-007-001)."
        )


@pytest.mark.human_review_required
@pytest.mark.contested(
    regulation="CIP-007-R2",
    reason="Industry interpretation varies on mitigation plan adequacy",
)
def test_CIP007_R2_mitigation_plan_adequacy(mitigation_plan):
    """
    This test does not assert adequacy. It verifies a current human
    determination of adequacy exists per ASSUME-007-002 criteria.
    """
    assert mitigation_plan.identifies_vulnerability, (
        f"INCOMPLETE: Mitigation plan {mitigation_plan.plan_id} does not "
        f"identify the specific vulnerability/CVE."
    )
    assert mitigation_plan.compensating_controls, (
        f"INCOMPLETE: Mitigation plan {mitigation_plan.plan_id} lists no "
        f"compensating controls."
    )
    assert mitigation_plan.owner_id, (
        f"INCOMPLETE: Mitigation plan {mitigation_plan.plan_id} has no owner."
    )
    six_months = timedelta(days=183)
    assert mitigation_plan.target_close_at is not None and \
           (mitigation_plan.target_close_at - mitigation_plan.created_at) <= six_months, (
        f"VIOLATION: Mitigation plan {mitigation_plan.plan_id} target close "
        f"date is more than 6 months from creation (ASSUME-007-002)."
    )
    assert mitigation_plan.last_reviewed_by_compliance_at and \
           mitigation_plan.last_reviewed_by_compliance_at > \
               datetime.utcnow() - timedelta(days=90), (
        f"STALE REVIEW: Mitigation plan {mitigation_plan.plan_id} has not "
        f"been reviewed by Compliance in the last 90 days."
    )
```

### `tests/CIP-007/test_R4.py` (Pattern 1)

```python
"""
CIP-007-6 R4 — Security Event Monitoring
Confidence: HIGH | Human Review: NOT REQUIRED
"""
from datetime import datetime, timedelta

REQUIRED_EVENT_TYPES = {
    "successful_login",
    "failed_login",
    "malicious_code_detected",
}


def test_CIP007_R4_event_types_logged(logged_assets):
    for asset in logged_assets:
        missing = REQUIRED_EVENT_TYPES - set(asset.event_types_captured)
        assert not missing, (
            f"VIOLATION: {asset.asset_id} not logging required event "
            f"types: {missing}"
        )


def test_CIP007_R4_review_cadence(log_review_records):
    """Reviews of logged events at intervals <= 15 calendar days."""
    by_asset = {}
    for r in log_review_records:
        by_asset.setdefault(r.asset_id, []).append(r.review_at)

    for asset_id, dates in by_asset.items():
        dates.sort()
        gaps = [(dates[i + 1] - dates[i]).days for i in range(len(dates) - 1)]
        for gap in gaps:
            assert gap <= 15, (
                f"VIOLATION: {asset_id} log review gap of {gap} days "
                f"exceeds 15-day threshold."
            )


def test_CIP007_R4_log_retention(logged_assets):
    for asset in logged_assets:
        assert asset.retention_days >= 90, (
            f"VIOLATION: {asset.asset_id} log retention is "
            f"{asset.retention_days} days (threshold 90)."
        )
```

### `tests/CIP-007/test_R5.py` (Pattern 2)

```python
"""
CIP-007-6 R5 — System Access Controls
Confidence: MEDIUM | Human Review: ASSUMPTION REQUIRED
"""
import pytest


@pytest.mark.assumption(
    id="ASSUME-007-003",
    description='"Technically feasible" exemption requires TFE filing + compensating controls',
    approved_by="Compliance Officer",
    review_date="2026-01-15",
)
def test_CIP007_R5_password_policy(hm_assets_with_interactive_access):
    for asset in hm_assets_with_interactive_access:
        if asset.tfe_on_file_for_password_policy:
            assert asset.compensating_controls_documented, (
                f"VIOLATION: {asset.asset_id} has TFE on password policy "
                f"but no compensating controls (ASSUME-007-003)."
            )
            continue

        policy = asset.password_policy
        assert policy.min_length >= 8, (
            f"VIOLATION: {asset.asset_id} password min_length="
            f"{policy.min_length} (threshold 8)."
        )
        assert policy.complexity_classes >= 3, (
            f"VIOLATION: {asset.asset_id} complexity_classes="
            f"{policy.complexity_classes} (threshold 3)."
        )
        assert policy.max_age_days <= 457, (
            f"VIOLATION: {asset.asset_id} max_age_days="
            f"{policy.max_age_days} (threshold 457)."
        )


def test_CIP007_R5_auth_failure_handling(hm_assets_with_interactive_access):
    for asset in hm_assets_with_interactive_access:
        has_threshold = asset.auth_failure_threshold is not None
        has_alert = asset.alert_on_auth_failure
        assert has_threshold or has_alert, (
            f"VIOLATION: {asset.asset_id} has neither a failed-auth "
            f"lockout threshold nor an alert configured."
        )
```

## Notes for the registry

- **R2 is the framework's prototype CONTESTED case.** The Pattern 3 test on mitigation-plan adequacy does not assert compliance; it asserts that a current, signed human determination exists. The substance of "is this compensating control good enough" is the auditor's call.
- **R5 TFE coupling:** Many older field devices physically cannot enforce 8-char/complex passwords. The TFE process is the legitimate escape valve but produces its own audit trail that must be tested for currency.
- **R4 review cadence pitfall:** Calendar days, not business days. A 15-day review gap that straddles a holiday weekend is still a violation.
