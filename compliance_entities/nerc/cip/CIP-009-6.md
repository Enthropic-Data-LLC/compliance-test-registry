# CIP-009-6 — Recovery Plans for BES Cyber Systems

**Registry path:** `/regulation-registry/NERC-CIP/CIP-009-6/`
**Version:** 6 (currently enforceable)
**Last parsed:** 2026-05-20
**Applies to:** Bulk Electric System (BES) owners, operators, and users in North America registered with NERC as a functional entity (Generator Owner/Operator, Transmission Owner/Operator, Balancing Authority, Reliability Coordinator, etc.) that own or operate BES Cyber Systems
**Trigger:** NERC registration as a functional entity + ownership/operation of BES Cyber Systems classified as High, Medium, or Low Impact per CIP-002; BES threshold: generation ≥ 20 MW, transmission ≥ 100 kV generally
**Jurisdiction:** North America — United States, Canada, and portions of Mexico; enforced by NERC and eight Regional Entities (WECC, SERC, MRO, RFC, NPCC, FRCC, TRE, SPP RE); FERC oversees NERC in the US
**Not applicable to:** Distribution-only utilities operating below the BES voltage threshold; behind-the-meter generation below 20 MW; entities not registered with NERC; natural gas pipeline operators (TSA SD-02D instead); water utilities (AWIA instead)
**Overall confidence:** HIGH — structurally parallel to CIP-008, with crisp cadences

---

## Scope summary

CIP-009 mandates recovery plans for High and Medium Impact BES Cyber Systems. The standard parallels CIP-008's structure: have a plan (R1), test it (R2), update after tests or invocations (R3). The key cadences are 15 calendar months for testing and 90 calendar days for post-test updates. Most failures here are cadence drift rather than substantive plan deficiencies.

"""
CIP-009-6 R2 — Recovery Plan Testing
Confidence: HIGH | Human Review: NOT REQUIRED
"""
from datetime import datetime, timedelta

WINDOW_15_MONTHS = timedelta(days=457)
WINDOW_36_MONTHS = timedelta(days=1096)


def _latest_test(records, of_type):
    matching = [r for r in records if r.test_type == of_type]
    return max(matching, key=lambda r: r.test_completed_at) if matching else None


def test_CIP009_R2_1_general_test_15mo(recovery_test_records):
    """R2.1 — any test type satisfies; just confirm one in 15 mo."""
    now = datetime.utcnow()
    if not recovery_test_records:
        assert False, "CRITICAL: No recovery plan test records exist."
    latest = max(recovery_test_records, key=lambda r: r.test_completed_at)
    age = now - latest.test_completed_at
    assert age <= WINDOW_15_MONTHS, (
        f"VIOLATION (R2.1): Latest recovery test {latest.record_id} is "
        f"{age.days} days old (threshold 457)."
    )


def test_CIP009_R2_2_backup_sample_15mo(recovery_test_records):
    """R2.2 — backup-sample restore test within 15 months."""
    latest = _latest_test(recovery_test_records, "backup_sample_restore")
    assert latest is not None, (
        "CRITICAL VIOLATION (R2.2): No backup-sample restore test on record."
    )
    age = datetime.utcnow() - latest.test_completed_at
    assert age <= WINDOW_15_MONTHS, (
        f"VIOLATION (R2.2): Backup sample test {latest.record_id} is "
        f"{age.days} days old (threshold 457)."
    )


def test_CIP009_R2_3_operational_exercise_36mo(recovery_test_records):
    """R2.3 — operational exercise within 36 months."""
    latest = _latest_test(recovery_test_records, "operational_exercise")
    assert latest is not None, (
        "CRITICAL VIOLATION (R2.3): No operational exercise on record."

    age = datetime.utcnow() - latest.test_completed_at

        f"VIOLATION (R2.3): Operational exercise {latest.record_id} is "
        f"{age.days} days old (threshold 1096)."
    )

## Notes for the registry

- **R2.2 vs R2.3 distinction is easy to miss.** R2.2 is a *sample backup restore* — verifying you can actually get data back. R2.3 is a *full operational exercise* — verifying you can rebuild a Cyber System end-to-end. Both have 15- and 36-month clocks respectively. Many entities treat R2.3 as satisfying R2.2; auditors do not.
- **No PARAMETERIZED elements.** The "representative sample" and "representative environment" language is open-ended, but those are scope decisions the auditor evaluates against the plan — not part of the deterministic test surface.
- **Backup-failure handling (R1):** The plan must address what happens when backups fail. The test verifies the section exists; substantive review of the procedure is manual.
