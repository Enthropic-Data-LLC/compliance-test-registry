# CIP-012-2 — Communications between Control Centers

**Registry path:** `/regulation-registry/NERC-CIP/CIP-012-2/`
**Version:** 2 (becomes enforceable 2026-07-01)
**Last parsed:** 2026-05-20
**Applies to:** Bulk Electric System (BES) owners, operators, and users in North America registered with NERC as a functional entity (Generator Owner/Operator, Transmission Owner/Operator, Balancing Authority, Reliability Coordinator, etc.) that own or operate BES Cyber Systems
**Trigger:** NERC registration as a functional entity + ownership/operation of BES Cyber Systems classified as High, Medium, or Low Impact per CIP-002; BES threshold: generation ≥ 20 MW, transmission ≥ 100 kV generally
**Jurisdiction:** North America — United States, Canada, and portions of Mexico; enforced by NERC and eight Regional Entities (WECC, SERC, MRO, RFC, NPCC, FRCC, TRE, SPP RE); FERC oversees NERC in the US
**Not applicable to:** Distribution-only utilities operating below the BES voltage threshold; behind-the-meter generation below 20 MW; entities not registered with NERC; natural gas pipeline operators (TSA SD-02D instead); water utilities (AWIA instead)
**Overall confidence:** MEDIUM — small standard but the "mitigation" language is PARAMETERIZED; cadences are DETERMINISTIC

> ⚠️ **Pre-enforcement standard.** CIP-012-2 becomes enforceable on July 1, 2026 (≈6 weeks from this parse). Currently enforceable predecessor is CIP-012-1. Tests below target v2; the registry's CI gate should not block on v2 violations until 2026-07-01.

---

## Scope summary

CIP-012 is the smallest CIP standard by line count — a single requirement (R1 in v1, expanded in v2) addressing the security of real-time Assessment and real-time monitoring data exchanged between Control Centers. Version 2 (Project 2020-04) extends the scope and clarifies what "Control Center" coverage means in the post-virtualization era. Applies to all impact levels, including Low — uncommon in the CIP suite.

The standard is intentionally outcome-oriented ("protect the confidentiality and integrity") rather than prescriptive (it does not mandate specific protocols), which is the main source of PARAMETERIZED elements.

---

## R1 — Plan to Protect Communication Links

### Source excerpt (R1)

> The Responsible Entity shall implement, except under CIP Exceptional Circumstances, one or more documented plan(s) to mitigate the risks posed by unauthorized disclosure and unauthorized modification of Real-time Assessment and Real-time monitoring data while being transmitted between any applicable Control Centers.

### Element extraction

| Element | Value | Classification |
|---|---|---|
| Subject | Real-time Assessment data and Real-time monitoring data transmitted between Control Centers | DETERMINISTIC (defined terms) |
| Condition | Data is in transit between Control Centers of the same or different Responsible Entities | DETERMINISTIC |
| Obligation | (R1.1) Identification of security protection used; (R1.2) Identification of where the Responsible Entity applied security protection for transmission of the data; (R1.3) If multiple entities are responsible, identification of the responsibilities of each Responsible Entity for applying security protection | PARAMETERIZED — "security protection" is not prescribed |
| Evidence | `cc_link_inventory[]`, `protection_method`, `responsibility_matrix`, `plan_approval_record` | PARAMETERIZED for protection method; DETERMINISTIC for plan presence |

### R2 (v2 NEW) — Implementation

In v1 this was implicit; v2 makes implementation an explicit requirement with the same outcome-orientation. The test surface is the same as R1 — verify the plan is implemented per its own terms.

## Generated tests

### `tests/CIP-012/test_R1.py` (Pattern 2)

```python
"""
CIP-012-2 R1 — Communications between Control Centers
Confidence: MEDIUM | Human Review: ASSUMPTION REQUIRED
Enforcement starts: 2026-07-01
"""
import pytest
from datetime import datetime

ENFORCEMENT_DATE = datetime(2026, 7, 1)

APPROVED_PROTECTION_METHODS = {
    "ipsec_aes256",
    "ipsec_aes128_gcm",
    "tls_1_2_mutual_auth",
    "tls_1_3_mutual_auth",
    "dedicated_mpls_with_carrier_attestation",
    "dedicated_leased_line_with_carrier_attestation",
}


def test_CIP012_v2_enforcement_window():
    """Soft fail before 2026-07-01; hard fail after."""
    if datetime.utcnow() < ENFORCEMENT_DATE:
        pytest.skip(
            f"CIP-012-2 enforcement begins {ENFORCEMENT_DATE.date()}. "
            f"Running tests in informational mode only."
        )


@pytest.mark.assumption(
    id="ASSUME-012-001",
    description="Mitigation = IPsec AES-256 / TLS 1.2+ mutual / dedicated link w/ attestation / approved alt",
    approved_by="Compliance Officer",
    review_date="2026-05-15",
)
def test_CIP012_R1_protection_method_documented(cc_communication_links):
    for link in cc_communication_links:
        assert link.protection_method is not None, (
            f"VIOLATION (R1.1): Link {link.link_id} between "
            f"{link.endpoint_a} and {link.endpoint_b} has no documented "
            f"protection method."
        )


def test_CIP012_R1_protection_method_approved(cc_communication_links):
    for link in cc_communication_links:
        if link.protection_method in APPROVED_PROTECTION_METHODS:
            continue
        # Approved-alternative escape valve per ASSUME-012-001(d)
        assert link.approved_alternative_record_id is not None, (
            f"VIOLATION: Link {link.link_id} uses "
            f"'{link.protection_method}' which is not on the approved "
            f"list and has no written approval record (ASSUME-012-001)."
        )


def test_CIP012_R1_application_point_identified(cc_communication_links):
    """R1.2: where on the link is protection applied?"""
    for link in cc_communication_links:
        assert link.protection_application_point, (
            f"VIOLATION (R1.2): Link {link.link_id} does not identify "
            f"where protection is applied (endpoint, mid-path device, etc.)."
        )


def test_CIP012_R1_3_multi_entity_responsibility(cc_communication_links):
    """R1.3: if both endpoints are different Responsible Entities, the
    responsibility-split must be documented and signed by both."""
    for link in cc_communication_links:
        if link.endpoint_a_entity == link.endpoint_b_entity:
            continue
        assert link.responsibility_matrix_id is not None, (
            f"VIOLATION (R1.3): Cross-entity link {link.link_id} "
            f"({link.endpoint_a_entity} <-> {link.endpoint_b_entity}) "
            f"has no responsibility matrix."
        )
        assert link.responsibility_matrix_signed_by_both, (
            f"VIOLATION (R1.3): Responsibility matrix {link.responsibility_matrix_id} "
            f"is not signed by both Responsible Entities."
        )


def test_CIP012_R1_link_inventory_complete(realtime_data_flows):
    """Every Real-time Assessment / Real-time monitoring flow must map to
    a CC link inventory entry."""
    for flow in realtime_data_flows:
        if not flow.endpoints_both_control_centers:
            continue
        assert flow.cc_link_id is not None, (
            f"VIOLATION: Real-time flow {flow.flow_id} between Control "
            f"Centers has no CIP-012 link inventory entry."
        )
```

## Notes for the registry

- **v2 effective date:** 2026-07-01. The `test_CIP012_v2_enforcement_window` fixture skips assertions before that date so this file can be merged immediately without false-positive failures.
- **Applies to all impact levels** — unusual. A Low impact Control Center with a real-time link to a Medium impact Control Center is fully in scope. This is the most-missed scoping question for entities that historically treated CIP-012 as an H/M-only concern.
- **Defined terms drift risk:** "Control Center" is being re-defined under FERC Order 919 (March 2026). When that definition takes effect, the `endpoints.both_are_control_centers` predicate may catch additional links. Watch for the implementation plan.
- **Cross-reference to CIP-005-7:** The encrypted-tunnel obligations here can overlap with ESP-to-ESP traffic protections. Avoid double-counting by treating CIP-012 as the authoritative source for inter-CC links and CIP-005 as authoritative for intra-CC ESP boundary traffic.
