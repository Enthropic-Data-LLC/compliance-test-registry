# CIP-008-6 — Incident Reporting and Response Planning

**Registry path:** `/regulation-registry/NERC-CIP/CIP-008-6/`
**Version:** 6 (currently enforceable)
**Last parsed:** 2026-05-20
**Overall confidence:** MIXED — R1, R3, R4 mostly DETERMINISTIC; R2 (plan implementation/testing) is HIGH; the substantive *classification* of an event as "Reportable" is PARAMETERIZED but the cadence and notification clock are DETERMINISTIC

---

## Scope summary

CIP-008 governs how Responsible Entities prepare for, identify, classify, respond to, and report cyber security incidents. The 2020 v6 revision added the "attempted compromise" reporting trigger and shortened the E-ISAC notification timeline. The cadences are crisp: 1 hour from determination for Reportable Cyber Security Incidents, 15 calendar months for plan review, 90 days for plan-update propagation.

"""
CIP-008-6 R4 — Notifications and Reporting
Confidence: MEDIUM | Human Review: ASSUMPTION REQUIRED
"""
import pytest
from datetime import timedelta

REQUIRED_RECIPIENTS = {"E-ISAC", "CISA"}


@pytest.mark.assumption(
    id="ASSUME-008-001",
    description="'Determination' = SIEM case classification by named analyst",
    approved_by="Compliance Officer",
    review_date="2026-01-15",
)
def test_CIP008_R4_reportable_1hr_notification(notifications):
    for n in notifications:
        if n.event_category != "Reportable":
            continue
        delta = n.sent_at - n.determined_at
        assert delta <= timedelta(hours=1), (
            f"CRITICAL VIOLATION: Reportable Incident {n.incident_id} "
            f"notification sent {delta.total_seconds() / 60:.0f} minutes "
            f"after determination (threshold 60)."
        )
        missing = REQUIRED_RECIPIENTS - set(n.recipients)
        assert not missing, (
            f"VIOLATION: Reportable Incident {n.incident_id} notification "
            f"missing recipients: {missing}"
        )


def test_CIP008_R4_attempted_compromise_end_of_next_day(notifications):
    for n in notifications:
        if n.event_category != "AttemptedCompromise":
            continue
        # End of next calendar day = determined_at.date() + 1 day, 23:59:59
        eod_next = (n.determined_at.replace(hour=23, minute=59, second=59)
                    + timedelta(days=1))
        assert n.sent_at <= eod_next, (
            f"VIOLATION: AttemptedCompromise {n.incident_id} notification "
            f"sent at {n.sent_at}, after end-of-next-day {eod_next}."
        )

## Notes for the registry

- **R4 is the most operationally sensitive** in the suite. A 1-hour clock means the determination workflow must produce an auditable timestamp the moment classification occurs. ASSUME-008-001 pins that timestamp to a system event, not a recollection.
- **v6 "attempted compromise" gotcha:** Many entities still under-report attempts because the threshold for "attempt" remains judgment-based at the analyst level. The test only verifies the clock once a determination is recorded — it does not (and cannot) verify the under-reporting risk.
- **R3 has hidden risk:** the 90-day clock starts at the *triggering event*, not at the *end of the test*. A 3-day tabletop exercise that ends 88 days before the deadline leaves no margin.
