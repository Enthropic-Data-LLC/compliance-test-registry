# CIP-003-9 — Security Management Controls

**Registry path:** `/regulation-registry/NERC-CIP/CIP-003-9/`
**Version:** 9 (enforceable as of 2026-04-01)
**Last parsed:** 2026-05-20
**Applies to:** Bulk Electric System (BES) owners, operators, and users in North America registered with NERC as a functional entity (Generator Owner/Operator, Transmission Owner/Operator, Balancing Authority, Reliability Coordinator, etc.) that own or operate BES Cyber Systems
**Trigger:** NERC registration as a functional entity + ownership/operation of BES Cyber Systems classified as High, Medium, or Low Impact per CIP-002; BES threshold: generation ≥ 20 MW, transmission ≥ 100 kV generally
**Jurisdiction:** North America — United States, Canada, and portions of Mexico; enforced by NERC and eight Regional Entities (WECC, SERC, MRO, RFC, NPCC, FRCC, TRE, SPP RE); FERC oversees NERC in the US
**Not applicable to:** Distribution-only utilities operating below the BES voltage threshold; behind-the-meter generation below 20 MW; entities not registered with NERC; natural gas pipeline operators (TSA SD-02D instead); water utilities (AWIA instead)
**Overall confidence:** MIXED — R1/R2 mostly DETERMINISTIC, Attachment 1 Sections 5–6 contain PARAMETERIZED language around "vendor electronic remote access" controls

---

## Scope summary

CIP-003 establishes overarching security management policies for both High/Medium and Low Impact BES Cyber Systems. The version-9 revision (effective April 1, 2026) adds Attachment 1 Section 6, which addresses vendor electronic remote access to Low Impact assets — historically the largest gap in CIP coverage.

CIP-003 is the standard most likely to produce PARAMETERIZED elements because it speaks in terms of policies, plans, and processes rather than discrete technical thresholds. The RDF response is to capture each policy domain as a presence-and-currency test, with the substantive content reviewed by humans.

## R2 — Low Impact BES Cyber System Plans (Attachment 1)

### Source excerpt

> Each Responsible Entity with at least one asset identified in CIP-002 containing low impact BES Cyber Systems shall implement one or more documented cyber security plan(s) for its low impact BES Cyber Systems that include the sections in Attachment 1.

Attachment 1 sections:
1. Cyber security awareness
2. Physical security controls
3. Electronic access controls (LERC — Low Impact External Routable Connectivity)
4. Cyber Security Incident response
5. Transient Cyber Asset and Removable Media malicious code risk mitigation
6. Vendor electronic remote access security controls *(new in v9)*

### `tests/CIP-003/test_R1.py` (Pattern 1)

```python
"""
CIP-003-9 R1 — Cyber Security Policies
Confidence: HIGH | Human Review: NOT REQUIRED
"""
from datetime import datetime, timedelta

REVIEW_WINDOW_DAYS = 457  # 15 calendar months

REQUIRED_HM_TOPICS = {
    "Personnel and training (CIP-004)",
    "Electronic Security Perimeters (CIP-005)",
    "Physical security of BES Cyber Systems (CIP-006)",
    "System security management (CIP-007)",
    "Incident reporting and response planning (CIP-008)",
    "Recovery plans for BES Cyber Systems (CIP-009)",
    "Configuration change management and vulnerability assessments (CIP-010)",
    "Information protection (CIP-011)",
    "Declaring and responding to CIP Exceptional Circumstances",
}

REQUIRED_LOW_TOPICS = {
    "Cyber security awareness",
    "Physical security controls",
    "Electronic access controls",
    "Cyber Security Incident response",
    "Transient Cyber Asset and Removable Media",
    "Vendor electronic remote access security controls",
}


def test_CIP003_R1_hm_policy_topics(hm_policy):
    """All 9 H/M policy topics covered."""
    missing = REQUIRED_HM_TOPICS - set(hm_policy.topics_covered)
    assert not missing, f"VIOLATION: H/M policy missing topics: {missing}"


def test_CIP003_R1_low_policy_topics(low_policy):
    """All 6 Low Impact policy topics covered (v9 includes vendor remote access)."""
    missing = REQUIRED_LOW_TOPICS - set(low_policy.topics_covered)
    assert not missing, f"VIOLATION: Low policy missing topics: {missing}"


def test_CIP003_R1_approval_cadence(hm_policy, low_policy):
    """CIP Senior Manager approval within 15 months."""
    now = datetime.utcnow()
    for policy in (hm_policy, low_policy):
        if policy is None:
            continue
        age = now - policy.approved_at
        assert age <= timedelta(days=REVIEW_WINDOW_DAYS), (
            f"VIOLATION: {policy.scope} policy approval is {age.days} days "
            f"old (threshold {REVIEW_WINDOW_DAYS})."
        )
        assert policy.approved_by_role in (
            "CIP Senior Manager",
            "CIP Senior Manager Delegate",
        ), (
            f"VIOLATION: {policy.scope} policy approved by "
            f"'{policy.approved_by_role}', not an authorized role."
        )
```

### `tests/CIP-003/test_R2.py` (Pattern 2 — Parameterized)

```python
"""
CIP-003-9 R2 — Low Impact BCS Plan, Attachment 1
Confidence: MEDIUM | Human Review: ASSUMPTION REQUIRED
"""
import pytest
from datetime import datetime, timedelta


@pytest.mark.assumption(
    id="ASSUME-003-001",
    description='"Necessary" communications = justified + on approved-flow list + enumerated pair',
    approved_by="Compliance Officer",
    review_date="2026-04-01",
)
def test_CIP003_R2_section3_lerc_controls(lerc_entry_points):
    """Attachment 1 Section 3: LERC inbound/outbound restricted to necessary."""
    for entry in lerc_entry_points:
        for flow in entry.permitted_flows:
            assert flow.justification, (
                f"VIOLATION: LERC entry {entry.entry_point_id} permits "
                f"{flow.protocol}/{flow.port} {flow.source}->{flow.destination} "
                f"with no documented justification (ASSUME-003-001)."
            )
            assert flow.on_approved_protocol_list, (
                f"VIOLATION: LERC entry {entry.entry_point_id} uses "
                f"{flow.protocol}/{flow.port}, not on approved-flow list."
            )


@pytest.mark.assumption(
    id="ASSUME-003-002",
    description="Section 6.2 disable capability = tested kill-switch within 15 months",
    approved_by="Compliance Officer",
    review_date="2026-04-01",
)
def test_CIP003_R2_section6_vendor_remote_access(low_impact_sites):
    """Attachment 1 Section 6 (NEW v9): Vendor remote access controls."""
    cutoff = datetime.utcnow() - timedelta(days=457)
    for site in low_impact_sites:
        if not site.has_vendor_remote_access:
            continue

        # 6.1 — determine sessions
        assert site.vendor_access_logging_enabled, (
            f"VIOLATION (6.1): Site {site.site_id} has vendor remote access "
            f"but no logging mechanism to determine active sessions."
        )

        # 6.2 — disable capability
        assert site.disable_capability_documented, (
            f"VIOLATION (6.2): Site {site.site_id} lacks documented "
            f"vendor-session disable capability (ASSUME-003-002)."
        )
        assert site.disable_capability_last_tested_at and \
               site.disable_capability_last_tested_at > cutoff, (
            f"STALE TEST (6.2): Site {site.site_id} disable capability "
            f"last tested {site.disable_capability_last_tested_at}. "
            f"Re-test required within 15 months."
        )

        # 6.3 — detection of malicious communications
        assert site.malicious_comm_detection_method, (
            f"VIOLATION (6.3): Site {site.site_id} has no documented "
            f"method to detect known/suspected malicious vendor comms."
        )
```

## Notes for the registry

- **Open assumptions:** 2 (ASSUME-003-001, ASSUME-003-002). Both require annual re-approval by Compliance Officer.
- **v9 transition risk:** Section 6 is brand-new as of April 2026. Most entities will fail their first audit on 6.2 (disable capability) because pre-v9 vendor access was typically managed by ad-hoc procedures. Test failures during the transition window should be treated as a tracked remediation backlog, not a re-test loop.
- **Cross-reference:** Section 6 overlaps with CIP-013-2 (Supply Chain Risk Management) for H/M assets. The same vendor session may be evaluated against both standards; ensure no duplicate test logic.
